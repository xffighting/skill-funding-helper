# 我重新整理了一个 Skill，让 AI Agent 也能记得原作者是谁

这两天我把 `Skill Funding Helper` 又重构了一遍。

一开始，它只是一个很直白的小工具。

帮 GitHub 上的 Skill 仓库补上打赏入口。

生成 `.github/FUNDING.yml`，补 README 里的支持区块，再顺手生成一份发布文案。

这个方向当然还能用。

但我后来越想越觉得，这个东西如果只面向人看 README，格局还是有点窄。

因为 Skill 真正被大量使用的场景，可能不是某个人专门打开仓库，认真读完所有说明。

更可能是 AI Agent 在执行任务时，发现自己需要某个能力，然后调用一个已经订阅，复制，安装，或者临时读取的 Skill。

那问题就来了。

Agent 用了这个 Skill 之后，怎么知道原作者是谁？

它该不该提示用户支持原作者？

如果这个 Skill 支持打赏，按次付费，付费支持，商业授权，Agent 应该怎么判断该展示哪条路径？

这才是我这次重构真正想解决的问题。

![封面图](../assets/wechat-cover.png)

以前我们讲开源项目，最常说的是代码。

一个函数，一个库，一个框架。

现在 Skill 有点不一样。

它开源的不是某段代码。

它更像一套工作方法。

比如怎么检查一个投标文件。

怎么复盘 GitHub issue。

怎么写外贸跟进邮件。

怎么把一个复杂流程拆成 Agent 可以执行的步骤。

这东西看起来是文字，其实里面有很多经验。

有判断顺序。

有边界感。

有踩过坑之后留下来的提醒。

如果一个 Skill 被很多 Agent 反复调用，它就不只是一个 prompt 文件了。

它会变成一个可复用的能力仓库。

能力仓库一旦开始被复用，就会遇到一个很现实的问题。

原作者怎么被看见？

原作者怎么获得一点继续维护的动力？

我不想把这件事讲得特别商业。

也不想把每个开源 Skill 都变成收费软件。

但我觉得，给原作者留一个体面的支持路径，是很正常的。

有人用爽了，愿意请作者喝杯咖啡。

团队把这个 Skill 接进自己的 Agent 工作流里，愿意买一次付费支持。

公司把它嵌进商业产品里，愿意谈一个商业授权。

这些都不丢人。

关键是入口要清楚，边界要清楚，Agent 也要读得懂。

所以这次我把项目核心改成了一个更适合 Agent 使用的结构。

中心不再只是 README 里的按钮。

而是一个 `skill-funding.json`。

![Agent 路由图](../assets/wechat-agent-route.png)

你可以把它理解成一份给 Agent 看的公开说明。

它回答几个问题。

谁是这个 Skill 的原作者。

Agent 在什么情况下需要署名。

用户想支持作者时，应该走哪条公开链接。

如果启用了按次收费，Agent 展示付款入口前需要先确认。

如果是商业使用，应该提示用户看授权说明。

同时，它还会把安全边界写清楚。

这个工具不处理支付。

不保存密钥。

不做静默计费。

不让 Agent 背着用户打开收费流程。

它只生成公开文件，让仓库，人，AI Agent 都能看懂。

这条线我很在意。

因为一旦涉及钱，尤其是 Agent 自动执行任务时，边界必须很清楚。

打赏可以轻。

授权可以谈。

按次收费也可以有。

但用户必须知道自己在做什么。

Agent 也必须先确认，再展示或引导付费路径。

这就是我想要的逻辑。

不是让 Skill 变得很商业。

而是让一个被复用的能力，拥有可解释，可维护，可追溯的支持方式。

![收益模型图](../assets/wechat-revenue-models.png)

这次我整理了四种收益路径。

第一种是打赏。

最轻。

用户觉得这个 Skill 帮到了自己，就通过 GitHub Sponsors，Ko-fi，Buy Me a Coffee，Open Collective 这类公开平台支持一下。

第二种是按次付费。

它更适合那种结果价值很明确的 Skill。

比如生成一次完整审查报告，做一次发布包，跑一次项目评估。

但这里一定要有显式确认。

Agent 不能偷偷把用户带去付款。

第三种是付费支持。

比如帮团队部署，改写 Skill，做接入，做培训，排查 Agent 工作流。

很多开源作者其实不一定想卖软件，但愿意卖服务。

这很合理。

第四种是商业授权。

如果有人把你的 Skill 放进自己的商业 Agent 产品里，或者作为客户交付的一部分长期使用，那就可以把授权边界写明白。

这不是挡住别人使用。

是把大规模商业使用和普通个人使用区分开。

我觉得未来很多高质量 Skill 都会走到这一步。

一个人把自己的经验沉淀成 Skill。

别人复制，安装，订阅，二次修改。

Agent 在工作流里调用它。

然后，当这个能力开始稳定创造价值时，原作者至少应该有机会被支持。

这个机会不一定很大。

但它应该存在。

![安全边界图](../assets/wechat-safety-boundary.png)

现在用起来也很简单。

初始化配置。

```bash
python3 scripts/skill_funding.py init --repo-root . --project-name 'My Useful Skill' --github your-github-user
```

生成资金入口，Agent manifest 和发布资料。

```bash
python3 scripts/skill_funding.py apply --repo-root . --config funding.config.json --update-readme --publish-kit --agent-manifest
```

查看 Agent 可以读取的支持卡片。

```bash
python3 scripts/skill_funding.py route --config funding.config.json --format markdown
```

检查仓库。

```bash
python3 scripts/skill_funding.py validate --repo-root .
```

跑完之后，仓库里会多出几类东西。

`skill-funding.json` 给 Agent 读。

`.github/FUNDING.yml` 给 GitHub Sponsor Button 用。

README 支持区块给人看。

`docs/publish-kit` 里放发布文案。

还有一个 Agent use card，用来解释这个 Skill 被订阅或复制后，应该如何署名和支持原作者。

我这次也顺手把项目介绍文档补成了中英文版本。

中文读者可以快速理解它解决什么问题。

英文读者也能直接拿去看项目定位，使用方式，收益模型和安全边界。

这对开源项目挺重要。

尤其是 Skill 这种东西，本身就很容易跨语言传播。

你写的是中文工作流，不代表只有中文用户会用。

你写的是英文 Skill，也不代表中文团队不能改造。

所以文档要尽量让更多人接得住。

我现在越来越相信，Agent 时代会出现很多很小，但很有价值的 Skill。

它们不一定酷。

不一定大。

甚至不一定需要复杂代码。

但它们能把某个人做事的路径，变成一套可以被复用的能力。

这件事很有意思。

以前开源的是代码。

后来开源的是模板和工作流。

现在我们开始开源能力。

当能力能被复制，它就需要署名。

当能力能创造价值，它就需要支持入口。

当能力被 Agent 调用，它就需要机器能读懂的规则。

这就是 `Skill Funding Helper` 这次升级后的方向。

不是一个单纯的打赏按钮生成器。

而是一套给 Skill 原作者准备的公开收益路由。

给人看，也给 AI Agent 看。

我觉得这个方向会继续往前走。

也许以后每个被频繁调用的 Skill，都会带着一份这样的 manifest。

上面写着作者是谁，怎么署名，怎么支持，怎么商业使用，哪些事情 Agent 不能做。

很小的一份文件。

但它把一个能力背后的人，重新放回了可见的位置。

这也是我这次重构最想留下的东西。
